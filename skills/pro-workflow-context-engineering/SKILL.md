# Context Engineering (上下文工程)

四个操作控制上下文如何在 AI 编程会话中流动。掌握它们就控制每一次回答的质量。

## 四大操作

### 1. Write — 把信息写到上下文之外

把信息移出上下文窗口，存入持久存储，这样压缩和会话边界后还能保留。

**写入位置：**

| 目标 | 时机 | 示例 |
|------|------|------|
| CLAUDE.md / AGENTS.md | 永久项目规则 | "始终使用 pnpm，不用 npm" |
| NOTES.md / scratchpad | 当前任务的工作状态 | 架构决策、开放问题 |
| `.learnings/` | 学习与模式 | 从纠正中来的 [LEARN] 规则 |
| 外部文件 | 太大无法放入上下文的数据 | 测试计划、迁移清单 |

**Scratchpad 工作流：**
```
1. 开始复杂任务 → 创建 NOTES.md 放目标和约束
2. 研究后 → 把发现写到 NOTES.md
3. 压缩后 → NOTES.md 存活，上下文不存活
4. 恢复 → 读 NOTES.md 恢复完整状态
```

### 2. Select — 检索相关信息

在合适的时间把合适的信息拉入上下文。精确比量更重要。

**按精确度排序：**

1. `@file` 引用 — 精确文件注入
2. `grep` / `Glob` — 定向模式搜索
3. 子代理探索 — 委托深度搜索
4. RAG / embeddings — 大代码库的语义检索

**核心原则：专注的 300 tokens > 无重点的 113K tokens。**

一个精准的 grep 返回正好需要的函数签名，比把整个模块丢进上下文更好。每个无关的 token 都稀释注意力。

**渐进式检索：**
```
1. 从文件名开始 (Glob)
2. 缩窄到特定函数 (Grep)
3. 只读相关行 (Read + offset+limit)
4. 从不读整个大文件如果只需要一个函数
```

### 3. Compress — 减少 tokens，保留信号

在不丢失重要信息的前提下缩小上下文。

**压缩策略：**

| 策略 | 方法 | 时机 |
|------|------|------|
| 聚焦压缩 | `/compact focus: auth 模块改动` | 任务边界 |
| 微压缩 | 让 AI 内联总结工具输出 | 大读取/搜索后 |
| 头+尾 | 读大输出的前20 + 后20行 | 日志分析、测试结果 |
| 工具结果清理 | 子代理结果报告后自动清除 | 重度探索 |
| 语义选择 | 总结发现，丢弃原始数据 | 研究阶段 |

**压缩触发点：**

- 规划后，执行前
- 完成一个功能或里程碑后
- 上下文超过 50%（设置 CLAUDE_AUTOCOMPACT_PCT_OVERRIDE=50）
- 切换任务领域前
- 重度搜索/读取操作后

### 4. Isolate — 跨执行空间分区

不要把所有东西放一个上下文。拆到独立执行空间。

| 方法 | 隔离级别 | 使用场景 |
|------|----------|----------|
| 子代理 | 分叉上下文 | 重度探索、测试运行、文档生成 |
| Worktrees | 完整仓库副本 | 并行功能、竞争方案 |
| `/btw` | 临时覆盖 | 快速问题不进入对话历史 |
| Agent 团队 | 独立会话 | 跨层级改动、并行审查 |
| 新会话 | 干净画布 | 无关工作、退化上下文 |

**子代理委托：**
```
主会话：规划、协调、提交
子代理1：探索 auth 模块，报告发现
子代理2：运行测试套件，报告失败
子代理3：生成迁移脚本
```

主上下文保持干净。子代理处理量。

## 上下文预算规划

示例基线：~200K 总窗口，~20K 开销（AGENTS.md、工具定义、MCP schema）。计划 **~180K 可用**。

| 分配 | 预算 | 放什么 |
|------|------|--------|
| 静态上下文 | 20-30K | AGENTS.md、工具 schema、MCP 定义 |
| 动态上下文 | 150-180K | 代码、对话、工具结果 |

**把静态上下文放前面。** AGENTS.md 和工具定义在对话前加载。保持稳定最大化 prompt 缓存命中 — 省钱省延迟。

| 阶段 | 目标使用 | 超标时行动 |
|------|----------|------------|
| 规划 | < 20% | 保持计划简洁，写到 scratchpad |
| 实现 | < 50% | 文件间压缩，委托读取 |
| 测试 | < 70% | 委托测试运行给子代理 |
| 审查 | < 85% | 如果退化就开新会话 |

## 何时 /clear vs /compact vs 子代理

| 情况 | 行动 |
|------|------|
| 任务边界，想保留学习 | `/compact` 加 focus |
| 上下文退化，AI 重复 | `/compact`，如果还不好就 `/resume` |
| 开始无关工作 | `/clear` 或新会话 |
| 重度读/搜操作 | 委托给子代理 |
| 快速附带问题 | `/btw` (不污染主上下文) |
| 探索多个方案 | Worktrees 或 agent 团队 |

## 反模式

- 只需要一个函数时加载整个文件
- 提取需要的信息后还留着 MCP 工具结果
- 运行 15+ MCP（每个都给每个请求添加工具 schema 开销）
- 模糊提示让 AI 广泛搜索 ("修一下代码")
- 从不压缩直到自动压缩在 95% 触发
