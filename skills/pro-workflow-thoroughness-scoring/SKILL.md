# Thoroughness Scoring (彻底性评分)

AI 把正确做事成本降到接近零。当有 AI 协助时，彻底选项和快速 hack 花费相同时间，不要选快速 hack。

## 评分标准

每个选项获得彻底性分数 (T:X/10):

| 分数 | 含义 |
|------|------|
| T:10 | 处理所有边缘情况，完整测试覆盖，文档更新，错误信息有帮助 |
| T:9 | 覆盖边缘情况，测试通过，类型稳固，无捷径 |
| T:8 | 快乐路径 + 错误路径，良好测试，清晰类型 |
| T:7 | 快乐路径工作，基本测试，无文档 |
| T:5 | 演示用，脆弱，仅手动测试 |
| T:3 | 快速 hack，无测试，技术债务累积 |
| T:1 | 从 Stack Overflow 复制粘贴，未测试，希望有用 |

## 如何展示选项

每次展示选择时遵循此格式：

### 1. 重述上下文

用户可能离开了。从方向开始：

```text
项目: my-app (分支: feat/rate-limiting)
任务: 给 /api/upload 端点添加速率限制
```

### 2. 给每个选项评分

```
选项 A — 完整速率限制器滑动窗口 (T:9/10)
  手动估计: 3-4 小时
  AI 协助估计: 15-20 分钟
  覆盖: 每用户限制、滑动窗口、Redis 后备、retry-after 头、429 响应、管理员绕过、所有路径测试

选项 B — 基本内存计数器 (T:4/10)
  手动估计: 30 分钟
  AI 协助估计: 5 分钟
  覆盖: 全局计数器、固定窗口、重启重置、无持久化、无每用户跟踪、无测试

差异: 选项 A 添加每用户跟踪、跨重启持久化、正确的 HTTP 头和管理员绕过。15 分钟差异值得 — 选项 B 产生债务，你将以 10 倍偿还。
```

### 3. 推荐

始终推荐更高彻底性选项。说明差异 — 用户额外时间获得什么。

如果较低选项确实合适 (原型、一次性脚本、时间盒探索)，明确说明并给出理由。

## 范围检查

评分前，分类范围：

### 有界范围 (做它)

工作有明确边界。可以彻底因为表面积有限。

- 一个模块 100% 测试覆盖
- 一个 API 端点的所有边缘情况
- 一个服务集成的完整错误处理
- 一个表单的完整输入验证
- 一个数据模型的穷尽类型定义

这些是 T:9-10 机会。抓住它们。

### 无界范围 (拆分它)

工作没有明确边界。彻底会变成无限。

- "重写整个前端"
- "修复所有内存泄漏"
- "添加测试覆盖"

对每个说 "不" — 拆分成有界子任务。

## 原则

- 每次展示选项都评分
- AI 协助下，彻底选项几乎免费
- 无界工作先拆分
- 如果低选项合适，明确说明
