# Orchestrate - 多阶段特性开发

通过有验证门的结构化阶段构建特性。

## 模式

```text
/develop <特性>
  │
  ├── 阶段 1: 研究 (编排 agent)
  │   └── 打分置信度 → GO/HOLD
  │
  ├── 阶段 2: 规划 (编排 agent)
  │   └── 展示计划 → 等待批准
  │
  ├── 阶段 3: 实现 (编排 agent)
  │   └── 执行计划 → 质量门
  │
  └── 阶段 4: 审查 (审查 agent)
      └── 代码审查 → 提交
```

## 使用方式

被要求构建特性时：

1. **从研究开始**: 委托给编排 agent 或 scout agent 探索代码库
2. **等待 GO/HOLD**: 如果置信度低于 70，不要继续
3. **展示计划**: 列出所有要改的文件、方法和风险
4. **获得批准**: 没有明确 "proceed" 不要实现
5. **逐步实现**: 每 5 次编辑过质量门
6. **提交前审查**: 在改动的审查运行审查 agent

## 何时使用

- 特性涉及 >5 个文件
- 需要架构决策
- 需求不清晰或复杂
- 跨领域关注点 (auth、日志、错误处理)
- 代码库中尚未建立的新模式

## 何时不使用

- 快速 bug 修复 (直接修)
- 单文件改动
- 理解的模式 (遵循现有代码)
- 仅文档改动

## Agent 选择

| 阶段 | Agent | 原因 |
|------|-------|------|
| 研究 | scout (后台、worktree) | 非阻塞探索 |
| 规划 | orchestrator (opus、memory) | 深度推理、模式回忆 |
| 实现 | orchestrator (opus、memory) | 完整工具访问 |
| 审查 | reviewer (读 + bash) | 安全和质量聚焦 |
| 调试 | debugger (opus、memory) | 系统调查 |

## 与 Pro-Workflow 集成

- 实现中的纠正触发自我修正循环
- 质量门在检查点通过 hooks 触发
- 学习在每个阶段结束时捕获
- 会话切换跨阶段工作
